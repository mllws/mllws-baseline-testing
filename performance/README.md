# performance/

Lighthouse CLI config plus a CrUX/PSI field-data pull for the 7 scoped URLs (homepage, About, Blog, Our Team, Contact, Volunteer, 404 page). See issue: "Set up performance baseline tooling."

- `lighthouse-config.json` — locked URL list, run count (3 per page/device), device list (`mobile`, `desktop`), Chrome launch config.
- `run-performance.js` — runs Lighthouse's `performance` category 3x per URL per device (mobile via Lighthouse's default throttled-mobile preset, desktop via Lighthouse's bundled `desktop-config.js`), saves every raw run JSON to `results/`, takes the median per metric, and writes the aggregated table to `../reports/performance-baseline.md`. Captures: performance score, LCP, INP, CLS, TTFB, FCP, page weight, request count. Run with `npm run perf:run`.
- `run-crux.js` — pulls PageSpeed Insights v5 (which wraps CrUX field data) per URL per device strategy, saves raw JSON to `results/`, and writes `../reports/crux-snapshot.md`. Works without an API key (shared rate limit); set `PSI_API_KEY` to use one. Pages/origins without enough real-user traffic have no CrUX data — recorded as "no CrUX data" rather than failing the run. Run with `npm run perf:crux`.

**WebPageTest cross-check — deferred, not scripted.** The plan calls for a WebPageTest API cross-check; it requires a personal API key from webpagetest.org that isn't provisioned yet. Not in this issue's acceptance criteria (Lighthouse config + Core Web Vitals capture + CrUX snapshot are). Add a `run-webpagetest.js` alongside these two if/when a key is available.

**INP note.** Lighthouse's lab-simulated `interaction-to-next-paint` audit needs a scripted interaction to produce a value — expect `N/A` on a plain page-load run. Treat the CrUX snapshot (`run-crux.js`) as the real INP source; it's field data from actual user interactions.
